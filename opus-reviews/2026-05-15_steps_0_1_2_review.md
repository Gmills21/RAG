## Step 2 — Add Docker Compose for Kotaemon

### What the PDR required (Section 9, Step 2)

**Local `docker-compose.yml`:**
- Use `ghcr.io/cinnamon/kotaemon:main-full`
- Bind to `127.0.0.1:7860:7860`
- `GRADIO_SERVER_NAME=0.0.0.0`, `GRADIO_SERVER_PORT=7860`
- Mount `./ktem_app_data` → `/app/ktem_app_data`
- Mount `./sample-data` → `/app/sample-data:ro`
- `extra_hosts` for `host.docker.internal:host-gateway`
- Comments explaining Ollama runs on the host
- No custom RAG service, no Postgres/pgvector/Qdrant

**Pilot `docker-compose.pilot.yml`:**
- Lightweight reverse proxy (Caddy)
- Only the proxy on 80/443
- Proxy routes to kotaemon on the private docker network
- Raw 7860 not exposed publicly
- `PUBLIC_HOST` and basic-auth env vars
- Single-customer, not multi-tenant

**`Caddyfile.example`:**
- Reverse proxy `PUBLIC_HOST` → `kotaemon:7860`
- Commented basic-auth instructions
- Marked single-customer-VPS only

Acceptance: `docker compose config` passes for both base and pilot, base
has exactly one `kotaemon` service, port mapping `127.0.0.1:7860:7860`,
host-gateway present, volume present, pilot has a reverse proxy, pilot
does not publicly expose 7860.

### What was actually delivered

Commit `8ddd0a4`. I ran the acceptance checks live:

## Cross-step cohesion observations

These are about how Steps 0/1/2 hand off to **each other** (not whether
each step individually followed the PDR).

### What flows well

- **Naming convention is consistent.** `*.example` for files the operator
  copies to a working file (`Caddyfile.example`, `.env.example`,
  `.env.pilot.example`). No mixed conventions.
- **`docker-compose.yml` ↔ `docker-compose.pilot.yml` separation is
  clean.** Local is the base; pilot is a pure override layer that adds
  Caddy and inherits everything else from base. This means Step 23's
  end-to-end run can use the same base file Step 2 produced.
- **Security posture is consistent across both modes.** Local binds to
  `127.0.0.1` only; pilot inherits that bind and only Caddy reaches the
  public internet. There is no path where raw 7860 is exposed.
- **Step 0's enforcement is real.** The `mvp-commander.mdc` rule cites
  `Overview/finalized_local_first_support_kb_pdr.md` (the actual PDR
  filename), not the placeholder `local_rag_mvp_pdr_cursor.md` from the
  prompt pack template. Someone actually edited it to match this repo,
  which is a good sign for cohesion going forward.
- **Step 1's placeholders exactly match the filenames Steps 3–22 will
  fill in.** I cross-referenced every placeholder against the PDR — no
  orphan placeholders, no missing destinations for upcoming steps.

### What was rough (now fixed)

- `.env.example` from Step 1 didn't know about the env vars that Step 2
  introduced into `docker-compose.yml`, so it was dead-on-arrival. Fixed.
- The Caddyfile from Step 2 had a runtime bug (log path) and a setup gap
  (no `cp Caddyfile.example Caddyfile` instruction). Fixed.
- Both compose files threw the deprecated-`version` warning on every
  invocation, which would also pollute Step 3+'s `preflight.sh` /
  `smoke_test_ollama.sh` outputs. Fixed.
- The `COMMANDER.md` step table broke its markdown rendering on row 3
  onward because pending rows were missing two columns. Fixed.
- `Overview/README.md` was a one-line stub. Fixed with a small navigation
  index.

### What I deliberately did NOT touch

- **`Makefile` stub** — wording mentions Step 6, which the PDR will fill.
  Borderline awkward but accurate; not a defect.
- **`restart: unless-stopped` on the local kotaemon service** — debatable
  for a laptop (auto-restarts on Docker daemon restart), but reasonable
  default for a long-running demo box. Leaving until the user gives a
  preference.
- **Retroactive PRs for Steps 0 and 1** — would require git/PR
  bookkeeping that is out of scope for a foundation review.
- **Anything that touches Step 3 (Ollama setup script)** — explicitly
  off-limits per your instruction.
