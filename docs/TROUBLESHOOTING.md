# Troubleshooting Guide

Use this guide when personal testing, a prospect demo, or a single-customer
pilot fails. Keep the profile boundary clear: Ollama is for `dev-ollama`
personal testing, while prospect demos and paid pilots default to hosted
inference for speed.

## Docker Not Running

Symptom: `docker compose up -d` fails with a daemon or connection error.

Likely cause: Docker Desktop or the Docker service is stopped.

Fix:

1. Start Docker Desktop or the Docker service.
2. Wait until Docker reports that it is running.
3. Re-run `docker compose up -d`.

Verification:

```bash
docker --version
docker compose version
docker compose ps
```

## Port 7860 Already In Use

Symptom: Kotaemon fails to start or Docker reports that port 7860 is already
allocated.

Likely cause: Another local app or an old Kotaemon container is using the port.

Fix:

1. Run `docker compose down`.
2. Stop the other process using port 7860, or change `KOTAEMON_PORT` in your
   shell for local testing.
3. Start again with `docker compose up -d`.

Verification:

```bash
docker compose ps
```

Manual check: open `http://localhost:7860` and confirm the Kotaemon UI loads.

## Ollama Not Installed For Personal Testing

Symptom: `ollama --version` returns `command not found` while using
`dev-ollama`.

Likely cause: Ollama is not installed or is not on `PATH`.

Fix:

1. Install Ollama from the official installer for your operating system.
2. Open a new terminal.
3. Run `./scripts/setup_ollama.sh` to pull the required local models.

Verification:

```bash
ollama --version
ollama list
```

## Ollama Installed But Server Not Running

Symptom: `curl http://localhost:11434/api/tags` fails, or Kotaemon cannot call
the local model while using `dev-ollama` or `pilot-private-gpu`.

Likely cause: The Ollama background server is not running.

Fix:

1. Start the Ollama desktop app, or run `ollama serve` in a separate terminal.
2. Confirm the API is reachable.
3. Run the smoke test.

Verification:

```bash
curl http://localhost:11434/api/tags
./scripts/smoke_test_ollama.sh
```

## Kotaemon Cannot Reach Ollama From Docker

Symptom: Kotaemon loads, but chat requests fail with connection errors to the
LLM or embedding endpoint.

Likely cause: Kotaemon runs inside Docker. From inside the container,
`localhost` points to the container itself, not your host machine.

Fix:

1. Keep Ollama running on the host.
2. In Kotaemon model resources, use:
   - Base URL: `http://host.docker.internal:11434/v1/`
   - API key: `ollama`
   - Chat model: `qwen3:4b`
   - Embedding model: `nomic-embed-text`
3. Start a new chat after saving model settings.

Verification:

```bash
docker compose ps
curl http://localhost:11434/api/tags
```

Manual check: ask a simple question in Kotaemon and confirm the response is
generated by the local model.

## Wrong Base URL: localhost vs host.docker.internal

Symptom: The model works from terminal smoke tests but fails inside Kotaemon.

Likely cause: Kotaemon is configured with `http://localhost:11434/v1/`.

Fix:

Use `http://host.docker.internal:11434/v1/` for resources configured inside the
Kotaemon container. Use `http://localhost:11434` only for host-side scripts such
as `scripts/smoke_test_ollama.sh`.

Verification:

Manual check: save the Kotaemon resource, start a new conversation, and ask
`Say "local model ok" and nothing else.`

## qwen3:4b Too Slow

Symptom: Answers work but take too long for a live demo.

Likely cause: Local laptop CPU/GPU resources are limited.

Fix:

1. Confirm the demo is pre-indexed before the call.
2. Use `qwen3:1.7b` for faster local development checks if quality is acceptable.
3. For external demos or pilots, use the hosted profile described in
   `docs/INFERENCE_PROFILES.md` instead of promising laptop-speed performance.

Verification:

```bash
ollama list
./scripts/smoke_test_ollama.sh
```

Manual check: ask one narrow support question and decide whether latency is
acceptable for the setting.

## Wrong Profile For A Demo Or Pilot

Symptom: A prospect demo or paid pilot is slow, and Kotaemon is configured with
`qwen3:4b`, `nomic-embed-text`, or `http://host.docker.internal:11434/v1/`.

Likely cause: The app is still using `dev-ollama`, which is only for personal
testing.

Fix:

1. For prospect demos, switch to `config/inference/demo-hosted.example.yml`.
2. For paid pilots, switch to `config/inference/pilot-hosted.example.yml`.
3. Re-index documents if the embedding model changes.
4. Start a new conversation after saving resources.

Verification:

Manual check: ask the seven questions in `docs/DEMO_SCRIPT.md`; answers should
be cited and responsive enough for a live call.

## Hosted Inference Fails

Symptom: `demo-hosted` or `pilot-hosted` answers fail with authentication,
quota, endpoint, or model errors.

Likely cause: Missing API key, wrong base URL, incorrect Azure deployment name,
or an endpoint not approved for the customer.

Fix:

1. Confirm `.env` or host secrets include the expected key.
2. Confirm the Kotaemon resource uses the profile values from `config/inference/`.
3. For Azure, use the deployment URL and deployment model name from that tenant.
4. Start a new chat after saving resources.

Verification:

Manual check: ask `Say "hosted profile ok" and nothing else.` Then ask a cited
sample-data question to confirm retrieval still works.

## Private GPU Base URL Is Wrong

Symptom: `pilot-private-gpu` is selected but Kotaemon cannot reach the model
server.

Likely cause: The model server is on the VPS host, but Kotaemon is running inside
Docker and was configured with `127.0.0.1`.

Fix:

1. If the model server runs on the VPS host, use `http://host.docker.internal:11434/v1/`.
2. If the model server runs as another Docker service, use that service URL.
3. Use `127.0.0.1` only when the model server is in the same container.

Verification:

Manual check: save the resource and ask `Say "private gpu profile ok" and
nothing else.`

## Embeddings Failing

Symptom: Upload or indexing fails, retrieval returns no useful sources, or
Kotaemon reports embedding errors.

Likely cause: the embedding model is missing, the API key is wrong, or the
embedding resource points to the wrong endpoint for the selected profile.

Fix:

1. Confirm the selected profile in `docs/INFERENCE_PROFILES.md`.
2. For `dev-ollama`, pull the embedding model with `ollama pull nomic-embed-text`.
3. For hosted profiles, confirm the API key, base URL, and embedding model.
4. Re-upload or re-index the affected documents.

Verification:

```bash
ollama list
curl http://localhost:11434/api/tags
```

Manual check: ask `Can we refund a customer after 20 days?` and confirm a source
from `sample-data/support/` is retrieved.

## Source Citations Not Appearing

Symptom: The answer text appears, but no sources or citations are visible.

Likely cause: The prompt pack is not applied, the collection was not selected,
or the answer came from model prior knowledge instead of retrieved context.

Fix:

1. Confirm the `support-demo` collection is selected.
2. Paste `prompt-packs/support_v1.md` into the relevant Kotaemon prompt setting.
3. Ask a question that is definitely covered by the sample corpus.
4. Open the source preview for at least one cited answer.

Verification:

Manual check: ask `What changed in the latest SSO release?` and confirm the
answer cites `04_release_notes.md`.

## Answers Hallucinating

Symptom: Kotaemon gives confident answers that are not supported by uploaded
docs.

Likely cause: The prompt is not strict enough, irrelevant docs were uploaded, or
conflicting/stale docs are polluting retrieval.

Fix:

1. Reapply `prompt-packs/support_v1.md`.
2. Remove stale, duplicate, or conflicting docs using
   `docs/SOURCE_QUALITY_CHECKLIST.md`.
3. Add missing source-of-truth docs instead of teaching the model from memory.
4. Use the refusal phrase `Not enough source support` when sources are missing.

Verification:

Manual check: ask `Is there a Berlin office lunch policy?` and confirm the bot
refuses instead of inventing an answer.

## Uploaded Docs Not Searchable

Symptom: Files appear uploaded, but answers do not use them.

Likely cause: Indexing is still running, parsing failed, the wrong collection is
selected, or the source file is too noisy.

Fix:

1. Wait for indexing to finish.
2. Confirm the chat is pointed at the collection that contains the files.
3. Convert messy Office/PDF/HTML files with `scripts/prepare_docs.py`.
4. Re-upload cleaner Markdown or CSV files.

Verification:

```bash
python scripts/prepare_docs.py --input sample-data/support --output tmp/prepared_support_docs
```

Manual check: open the uploaded document in Kotaemon source preview.

## Need To Reset All Local Data

Symptom: The local app is in a confusing state, old collections are interfering,
or you need a clean demo environment.

Likely cause: Old Kotaemon state remains in `./ktem_app_data`.

Fix:

1. Run the safe reset helper.
2. Start Kotaemon again.
3. Re-upload only the approved demo or customer docs.

Verification:

```bash
bash -n scripts/reset_local_data.sh
./scripts/reset_local_data.sh --yes
ls -ld ktem_app_data
docker compose up -d
```

Manual check: open `http://localhost:7860`, recreate the collection, and upload
the intended docs only.
